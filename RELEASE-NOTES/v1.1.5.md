# @everystate/view v1.1.5 Release Notes

**Released:** March 21, 2026
**Type:** Minor feature release

This release adds five major features that significantly expand everystate-view's capabilities, transforming it from a basic declarative renderer into a production-ready UI framework.

---

## New Features

### 1. Component Escape Hatch

**Imperative component mounting for complex widgets and third-party library integration.**

The `mount()` function now accepts a `components` parameter that allows you to register custom mount functions for nodes marked with a `component` property.

**API:**
```js
mount(store, prefix, container, handlers, components)
```

**Usage:**
```js
const components = {
  codeEditor: (store, el) => {
    const editor = CodeMirror(el, { /* config */ });
    editor.on('change', () => store.set('code', editor.getValue()));
    return () => editor.toTextArea(); // teardown
  }
};

mount(store, 'ui', document.body, {}, components);
```

**In your view spec:**
```js
{
  tag: 'div',
  component: 'codeEditor',
  class: 'editor-container'
}
```

**Benefits:**
- Integrate third-party libraries (charts, editors, maps) without fighting the framework
- Full control over imperative DOM manipulation when needed
- Clean teardown via returned function
- Works in both regular nodes and forEach templates

### 2. Drag-and-Drop Event Handlers

**Full HTML5 drag-and-drop API support.**

Five new event handlers for building drag-and-drop interfaces:

- `onDragStart` - fired when drag begins
- `onDragEnd` - fired when drag completes
- `onDragOver` - fired continuously while dragging over element
- `onDragLeave` - fired when drag leaves element
- `onDrop` - fired when item is dropped

**Usage:**
```js
{
  tag: 'div',
  draggable: true,
  class: 'task-card',
  text: 'Drag me',
  onDragStart: 'handleDragStart'
}
```

**Handler example:**
```js
const handlers = {
  handleDragStart: (ctx, e) => {
    e.dataTransfer.effectAllowed = 'move';
    e.dataTransfer.setData('text/plain', ctx.item.id);
  },
  handleDrop: (ctx, e) => {
    e.preventDefault();
    const id = e.dataTransfer.getData('text/plain');
    // Update state to move item
  }
};
```

Available on both regular nodes and forEach template specs.

### 3. Fragment forEach (No-Wrapper Expansion)

**Render forEach items inline without a wrapper element.**

Previously, forEach loops required a container element. Now you can expand items directly into the parent by omitting the `tag` property.

**Before (with wrapper):**
```js
{
  tag: 'ul',
  forEach: 'items',
  as: 'item',
  template: { tag: 'li', text: '{{item.name}}' }
}
```

**After (fragment):**
```js
{
  forEach: 'items',  // No tag!
  as: 'item',
  template: { tag: 'li', text: '{{item.name}}' }
}
```

**How it works:**
- Uses HTML comment markers (`<!--fragment-start-->`, `<!--fragment-end-->`) to track boundaries
- Items render directly into parent container
- Fully reactive: re-renders when the data array changes
- No extra DOM nodes

**Use cases:**
- Table rows without wrapping `<tbody>`
- List items without wrapping `<ul>`
- Inline text spans without wrapping `<div>`

---

### 4. Nested forEach in Templates

**Iterate over nested data structures with forEach inside forEach.**

You can now use forEach loops inside template specs, enabling nested iteration patterns.

**Example:**
```js
{
  tag: 'div',
  forEach: 'categories',
  as: 'category',
  template: {
    tag: 'section',
    children: [
      { tag: 'h2', text: '{{category.name}}' },
      {
        forEach: 'category.items',  // Nested forEach!
        as: 'item',
        template: { tag: 'p', text: '{{item.title}}' }
      }
    ]
  }
}
```

**Features:**
- Supports arbitrary nesting depth
- Custom `as` naming for each level
- Reactive updates on nested data changes
- Uses comment markers for efficient boundary tracking

---

### 5. Enhanced `bind` Semantics

**Improved two-way binding with broader element support.**

The `bind` property now works intelligently across different element types:

**Two-way binding (inputs):**
- `<input>` (all types including checkbox)
- `<textarea>`
- `<select>`

**One-way binding (display):**
- Any other element (e.g., `<span>`, `<div>`, `<p>`)
- Updates `textContent` instead of `value`

**Example:**
```js
{
  tag: 'input',
  bind: 'user.name'  // Two-way: typing updates state
}
```

```js
{
  tag: 'span',
  bind: 'user.name'  // One-way: displays state, no input event
}
```

**Improvements:**
- Cleaner conditional logic distinguishing input vs. display elements
- Checkbox handling preserved
- Cursor position preserved during updates (no jump on external changes)

## Migration Guide

This release is **backward compatible**. No breaking changes.

**Optional upgrades:**

1. **If you need custom components:** Add a `components` object to your `mount()` call.
2. **If you need drag-and-drop:** Add `draggable: true` and drag event handlers to your specs.
3. **If you have unwanted wrapper elements:** Remove the `tag` property from forEach nodes to use fragments.
4. **If you need nested iteration:** Add forEach specs inside template children.
5. **If you want display binding:** Use `bind` on non-input elements.
